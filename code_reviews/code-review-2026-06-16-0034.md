## Findings

### [P1] Search results retain and render full content-heavy lines

`Searcher.parseResults` stores the entire matching line in every `SearchResult`, and the app keeps up to 500 of those values in `DocumentViewModel.searchResults`. For the current real-world `before_provider_request` records, a single line can be hundreds of KB. A broad search can therefore retain hundreds of MB in the view model before the user clicks anything. `SearchResultsView` then builds a `Text` value from the full line, splitting it into full `before`/`match`/`after` strings even though the UI only shows two visual lines.

- [Sources/BigJSONLCore/Search/Searcher.swift:145](../Sources/BigJSONLCore/Search/Searcher.swift#L145)
- [Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift:298](../Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift#L298)
- [Sources/BigJSONLApp/Views/SearchResultsView.swift:52](../Sources/BigJSONLApp/Views/SearchResultsView.swift#L52)

Store bounded snippets rather than full match lines for UI search results, or make `SearchResult.lineText` optional/truncated for capped searches. The renderer should highlight a small window around the first match instead of constructing `Text` from the entire line.

#### Resolution — Fixed

Search parsing now stores a bounded snippet around the first literal match, or a bounded prefix when the grep/ripgrep match came from a pattern that is not found literally in the output line. `Search results retain bounded snippets instead of full lines` passed, confirming a 40 KB matching line is retained as a sub-500-character result snippet.

### [P1] Viewport line validation still runs synchronously on the main actor

The heavy inspector path was moved off the main actor, but normal viewport loading still reads, decodes, trims, and validates every visible line synchronously on `DocumentViewModel`, which is `@MainActor`. A taller window now increases `viewportLineCount`, so opening, resizing, scrolling, and jumping can parse many content-heavy JSON records on the UI actor. This can reproduce visible hangs even if the selected inspector content prepares in the background.

- [Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift:9](../Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift#L9)
- [Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift:127](../Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift#L127)
- [Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift:143](../Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift#L143)
- [Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift:214](../Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift#L214)

Move viewport preparation to a cancellable background task, publish the resulting `[LineInfo]` back to the main actor, and consider deferring JSON validation until a row is selected or until an idle/background pass finishes. If invalid-row tinting must remain immediate, cap or time-slice validation so resizing and scrolling stay interactive.

#### Resolution — Fixed

Viewport loading now runs through a generation-checked detached task. The background loader indexes, reads, decodes, and validates rows, then publishes the completed window back to the main actor. The app tests now await asynchronous loading; `Viewport advances and retreats with bounded overlapping windows` and `Viewport grows to fill a taller line pane` both passed.

### [P2] Indexing copies every scanned chunk out of the mmap

`LineOffsetIndex.ensureLineIndexed` reads a no-copy `DispatchData` slice from `MappedFile`, immediately copies it into `Data`, and then copies again into `[UInt8]` before scanning for newline bytes. Jumping far into a multi-GB file or indexing to EOF therefore allocates and copies every scanned byte, even though the file is already memory-mapped.

- [Sources/BigJSONLCore/Indexing/LineOffsetIndex.swift:86](../Sources/BigJSONLCore/Indexing/LineOffsetIndex.swift#L86)
- [Sources/BigJSONLCore/Indexing/LineOffsetIndex.swift:90](../Sources/BigJSONLCore/Indexing/LineOffsetIndex.swift#L90)
- [Sources/BigJSONLCore/FileIO/MappedFile.swift:64](../Sources/BigJSONLCore/FileIO/MappedFile.swift#L64)

Expose a safe `withUnsafeBytes(offset:length:)` API on `MappedFile`, or make `LineOffsetIndex` scan `DispatchData` regions without materializing an array. That keeps indexing proportional to page faults and CPU scanning instead of adding large transient heap traffic.

#### Resolution — Fixed

`MappedFile` now exposes scoped mapped-byte access, and `LineOffsetIndex.ensureLineIndexed` scans those bytes directly instead of converting each chunk through `Data` and `[UInt8]`. `MappedFile reads bytes from known offset` covers the new direct access path, and the existing index tests passed.

### [P2] Closing or reloading tabs does not cancel old work or release all resources

When closing a tab in a multi-tab window, the tab is removed from the array without stopping its security-scoped resource or cancelling any in-flight search/inspector work. Closing the final tab stops the security scope and clears the document, but leaves the old `viewModel` attached to the now-empty `TabItem`. Reloading also replaces the document and view model without cancelling work on the old view model. Those old tasks can keep mapped files, search subprocesses, inspector caches, and large formatted strings alive after the user believes the tab is gone.

- [Sources/BigJSONLApp/BigJSONLApp.swift:93](../Sources/BigJSONLApp/BigJSONLApp.swift#L93)
- [Sources/BigJSONLApp/BigJSONLApp.swift:97](../Sources/BigJSONLApp/BigJSONLApp.swift#L97)
- [Sources/BigJSONLApp/BigJSONLApp.swift:105](../Sources/BigJSONLApp/BigJSONLApp.swift#L105)
- [Sources/BigJSONLApp/TabItem.swift:39](../Sources/BigJSONLApp/TabItem.swift#L39)

Add a `TabItem.close()`/`dispose()` path that cancels the view model, clears it, and stops security-scoped access. Call it for multi-tab close, last-tab reset, and before replacing a view model during reload/open.

#### Resolution — Fixed

`TabItem` now disposes old view models before open/reload/close, and `DocumentViewModel.dispose()` cancels viewport/search/inspector work, releases the mapping, and clears retained row/search/inspector state. Multi-tab close and last-tab reset both call the close path. `Disposing the view model cancels work and clears retained content` passed.

### [P2] Stale inspector formatting work is discarded only after it finishes

`prepareInspector` cancels the parent `inspectorTask`, but the expensive formatter runs inside `Task.detached`. Detached tasks do not inherit cancellation from the parent task, so rapidly selecting several large lines can leave multiple JSON formatting/tokenizing jobs running concurrently. The stale-result guard prevents old content from being shown, but it does not prevent the CPU and memory work from happening.

- [Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift:228](../Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift#L228)
- [Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift:245](../Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift#L245)
- [Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift:250](../Sources/BigJSONLApp/ViewModel/DocumentViewModel.swift#L250)

Keep a handle to the detached formatting task and cancel it directly, or move formatting into a nonisolated async helper that cooperatively checks cancellation while formatting/tokenizing. The formatter/tokenizer loops should accept cancellation checks at structural boundaries for very large records.

#### Resolution — Fixed

The view model now keeps and cancels the detached inspector formatting task directly. `JSONFormatter.displayContentCancellable` and the tokenizer's cancellable valid-JSON path check cancellation during large formatting/tokenization loops, so rapid selection changes can stop stale work instead of only discarding its result. `Inspector content prepares asynchronously and reuses its cache` continued to pass.
